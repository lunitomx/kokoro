# Story: Modelo de datos para Client Graph

## User Story
As Kokoro (Eduardo's coaching system),
I want typed Pydantic models for clients (ClientProfile, ClientRegistry),
so that client knowledge can be structured, validated, and persisted.

## Acceptance Criteria

### Scenario: Create a ClientProfile
```gherkin
Given a client name "Konecta Park" with group "invertikal"
When a ClientProfile is created
Then it has typed fields for name, group, repos, segments, industry, metadata
And all fields are validated by Pydantic
```

### Scenario: ClientRegistry holds multiple clients
```gherkin
Given two ClientProfiles (Konecta Park, Escuela Libre)
When added to a ClientRegistry
Then registry.clients contains both
And find_by_name("konecta") returns the Konecta Park profile
```

## Examples

| Input | Expected |
|-------|----------|
| ClientProfile(id="konecta-park", name="Konecta Park", group="invertikal") | Valid model |
| ClientRegistry with 2 clients, find_by_name("konecta") | Returns konecta-park |
| ClientProfile without required name | ValidationError |

## Notes
- Separate module: src/kokoro/clients/ (not in ontology/)
- Follow same patterns as ontology/models.py (Pydantic, StrEnum, type annotations)
- Pyright strict compatible
