# S32.2 — Story Design: /kokoro-landing Skill Command

## Problem

No tool exists to audit a landing page as a **decision sequence**. The current
skill set has `/kokoro-audit` for technical health (SEO, performance, a11y) and
`/kokoro-launch` for generating landing page copy, but nothing evaluates the
strategic conversion flow — whether the right blocks are present, whether the
copy califies or convinces, whether the decision sequence follows the correct
psychological order.

## Value

Entrepreneurs get actionable feedback on their landing page's decision
architecture before investing in traffic. Instead of learning post-spend that
their landing leaks qualified leads, they get a structured audit that maps to
the 9-block framework and 5 principles from the Lean Landing Page methodology.

## Approach

Create a single command file following the established `kokoro-creative-review.md`
pattern. The file:

1. References `kokoro-lean-landing.md` as its knowledge source
2. Provides a 5-step guided audit process (obtain, map, evaluate, scorecard, diagnosis)
3. Produces structured output: block map table, principle scorecard, per-block diagnosis, top 3 actions
4. Uses Eduardo's voice throughout (vocabulario luxurizante, Proyector strategy, amortiguar-pivotar-ofrecer)
5. Is mirrored identically to both command locations

### Pattern Reference

The command file structure follows `kokoro-creative-review.md` exactly:

```
Title > tagline > Eduardo quote
  Contexto (reads knowledge file)
  Diferencia con otros skills
  Resolucion de invitado (standard pattern from clients.json)
  Estrategia del Proyector (wait for invitation)
  Paso a paso (5 steps)
  Output estructurado
  Anti-patrones
  Persistencia
```

### 5-Step Process

| Step | Name | Input | Output |
|------|------|-------|--------|
| 1 | Obtener landing | URL (WebFetch), pasted HTML, or consultive mode | Raw content to analyze |
| 2 | Mapeo de bloques | Raw content + 9 block definitions from knowledge | Block map table (present/absent/partial) |
| 3 | Evaluacion por bloque | Present blocks + reglas de oro | Per-block score (fuerte/parcial/debil) + copy actual + rewrite sugerido |
| 4 | Scorecard de principios | Full landing analysis + 5 principles | Principle scorecard (cumple/parcial/no cumple + evidencia) |
| 5 | Diagnostico integrado | All prior analysis | Missing blocks, disorder, top 3 actions, /kokoro-audit suggestion |

## Output Structure

### Block Map Table

```
## Mapa de Bloques

| # | Bloque | Estado | Ubicacion/Nota |
|---|--------|--------|----------------|
| 01 | HERO | Presente | Seccion principal con headline + CTA |
| 02 | PAINS | Ausente | — |
| 03 | INERCIAS | Parcial | Hay FAQ pero no aborda bloqueos de decision |
| 04 | ASI FUNCIONA | Presente | Proceso de 4 pasos con timeline |
| 05 | MAFIA OFFER | Ausente | — |
| 06 | NO ES PARA TI SI | Ausente | — |
| 07 | FRICTIONS | Parcial | Logos presentes pero sin contexto ni resultados |
| 08 | QUID PRO QUO | Presente | Formulario con calendario |
| 09 | CTA PERSISTENTE | Parcial | CTA solo aparece en hero y footer |

Resumen: 3/9 presentes, 3/9 parciales, 3/9 ausentes
```

### Principle Scorecard

```
## Scorecard de Principios

| Principio | Estado | Evidencia |
|-----------|--------|-----------|
| Claridad en 10-20s | Cumple | Headline comunica PUV clara con resultado medible |
| Riesgo invertido | No cumple | Sin garantia, sin mafia offer, sin compromiso audaz |
| Filtro de prospectos | No cumple | No hay seccion "No es para ti si..." |
| Proceso visible | Cumple | 4 pasos con timeline y entregables |
| CTA unico y persistente | Parcial | CTA consistente pero solo en 2 posiciones |
```

### Per-Block Diagnosis

```
## Diagnostico por Bloque

### 01. HERO
Score: parcial
Copy actual: "Bienvenido a nuestra empresa de automatizacion..."
Rewrite sugerido: "Estas por perder un proyecto ya que tus pruebas de fuga son pobres — y lo peor es que tu equipo ya lo sabe..."
Nota: El hero no usa evento detonante. El headline es generico ("bienvenido") en lugar de reflejar el dolor inmediato del ICP.
```

### Top 3 Actions

```
## Top 3 Acciones Inmediatas

1. **Agregar bloque MAFIA OFFER** — Sin riesgo invertido, el prospecto no tiene razon para bajar el escudo. Crear una condicion fuerte + beneficio claro.
2. **Reescribir HERO con evento detonante** — El headline debe activar reconocimiento inmediato: "eso me esta pasando a mi."
3. **Agregar seccion NO ES PARA TI SI** — El filtro eleva el estatus y pre-califica. Minimo 3 criterios de exclusion.
```

## Decisions

| ID | Decision | Rationale |
|----|----------|-----------|
| D1 | Follow `kokoro-creative-review.md` pattern exactly | Established pattern from E31 — reduces cognitive load for maintainers and ensures consistency across all Kokoro skills |
| D2 | 3 input modes: URL, pasted HTML, consultive | WebFetch handles most cases, pasted HTML covers SPAs/JS-rendered pages, consultive mode allows evaluation of described structures without a live page |
| D3 | Block scoring uses fuerte/parcial/debil (not numeric) | Consistent with qualitative evaluation style of Eduardo's methodology — the goal is diagnosis, not a score to optimize |
| D4 | Suggest `/kokoro-audit` at end, not beginning | Avoid confusion about the difference between strategic and technical evaluation — let the user complete the strategic audit first |

## Integration Points

None. This is a standalone command file that reads a knowledge file. No Python code, no API calls, no external dependencies beyond WebFetch (which is a Claude Code built-in).

## Risks

| Risk | Mitigation |
|------|------------|
| WebFetch fails on SPA/JS-rendered pages | Documented fallback: ask for pasted HTML or screenshot |
| User confuses with `/kokoro-audit` | Explicit "Diferencia con otros skills" section in the command file |
