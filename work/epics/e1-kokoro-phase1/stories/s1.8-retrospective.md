# S1.8 Retrospective — Meta Skills (/kokoro + /kokoro-session)

**Story:** S1.8 Meta Skills (/kokoro + /kokoro-session)
**Size:** M (estimated) → M (actual)
**Date:** 2026-03-09
**Estimated:** 50 min | **Actual:** ~50 min

## Summary

Final story in E1 (Kokoro Phase 1). Created two meta-skills: `/kokoro` (router
that diagnoses where the entrepreneur is in Phase 1 and recommends the right
next skill) and `/kokoro-session` (session manager with start/continue/review
modes). Also fixed the CLI prefix match from `startswith("kokoro-")` to
`startswith(("kokoro-", "kokoro."))` to ensure `kokoro.md` is treated as
Kokoro-owned on reinstall. 31 new tests across 2 test files. QR caught the
prefix fix being too broad (`startswith("kokoro")` would match unintended files)
— tightened to tuple match. 159/159 tests GREEN, pyright 0 errors, ruff clean.
5 commits (T1-T4 + QR fix). M3 Feature Complete achieved — all 8 stories done.

## What Went Well

- **QR caught a real defect in the CLI fix** — The original design specified
  `startswith("kokoro")` which would match any file starting with "kokoro"
  (e.g., a hypothetical `kokoroville.md`). QR identified this and the fix was
  tightened to `startswith(("kokoro-", "kokoro."))` — a precise tuple match
  that covers exactly `kokoro-*.md` and `kokoro.md`. This is the second
  consecutive story where QR caught a real issue (S1.7: weak assertion, S1.8:
  overly broad prefix).

- **Meta-skill pattern emerged** — The router skill introduces a new pattern:
  skills that reference other skills without executing them. This "meta-skill"
  pattern is distinct from PAT-L-001 (content-only skill) because the router
  must know about all 4 Phase 1 skills by name, purpose, and order. This
  awareness creates a coupling that content-only skills do not have.

- **Two skill files in one story worked** — Despite being more complex than
  S1.4-S1.7 (2 files + CLI modification vs 1 file + 0 CLI changes), the M
  sizing was accurate. The PAT-L-001 pattern provided enough predictability
  to absorb the extra scope.

- **E1 complete** — 8 stories, 3 milestones, all GREEN. The epic went from
  package skeleton to a complete Phase 1 guided experience with 4 methodology
  skills + 2 meta-skills + knowledge files + CLAUDE.md brain.

## What to Improve

- **Design precision on CLI changes** — The design specified
  `startswith("kokoro")` which was technically correct but semantically too
  broad. The QR step caught this, but a more precise design ("match files
  named `kokoro.md` or starting with `kokoro-`") would have prevented the
  issue entirely. For future stories: when modifying string matching logic,
  specify the exact match semantics, not just the code change.

- **Test file consolidation opportunity** — With 8 test files and 159 tests,
  there may be value in shared fixtures or a conftest.py for common patterns
  (skill file loading, content search). Not urgent, but worth considering for
  Phase 2.

## Heutagogical Checkpoint

1. **What did I learn?** Meta-skills (skills that reference other skills) have
   a fundamentally different coupling profile than content-only skills. A router
   must know about all referenced skills — their names, order, and purpose. This
   creates a maintenance burden: adding a new Phase 1 skill would require
   updating the router AND the session manager. This is acceptable for a small
   set (4 skills) but would not scale to 20+ skills without a registry pattern.

2. **What would I change about the process?** The CLI prefix fix was a 1-line
   change but went through design -> test -> implement -> QR fix — 4 touches
   for 1 line. The process was correct (TDD caught nothing, but QR caught a
   real defect), but I would front-load semantic review of string matching
   changes in the design phase. A simple table of "what matches / what should
   not match" would have caught the `startswith("kokoro")` issue before code.

3. **Are there improvements for the framework?** Two observations:
   - QR has now caught real defects in 2 consecutive stories (S1.7, S1.8).
     This validates QR as non-optional even for pattern-following stories.
   - The "meta-skill" pattern (router/session manager) is distinct enough from
     PAT-L-001 to deserve its own pattern identifier. It involves: awareness
     of other skills, methodology ordering, diagnostic flow, and no direct
     execution of referenced skills.

4. **What am I more capable of now?** Designing meta-skills — skills that
   orchestrate other skills without executing them. I understand the coupling
   implications (router must update when skills change) and the design
   constraints (ask-don't-execute, Projector strategy, no persistent state).
   I also have stronger evidence that QR is load-bearing in the process, not
   ceremony.
