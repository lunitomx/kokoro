## Execution Order

1. **T1** — Tests first (TDD RED). Defines the contract for both skill files and the CLI fix.
2. **T2** — CLI fix + router skill (TDD GREEN). CLI fix is 1 char and unblocks the overwrite test. Router is the diagnostic entry point.
3. **T3** — Session skill (TDD GREEN). Independent from T2 content-wise, but sequenced after for clean commits. Could technically parallelize with T2.
4. **T4** — Integration validation. Confirms both files are copied by CLI and full suite is clean.

**Rationale:** T1 locks down all requirements as executable tests. T2 bundles the CLI fix with the router skill because the overwrite test for `kokoro.md` depends on the fix. T3 is a separate creative task for the session skill. T4 is the integration gate. Risk-first: the CLI fix (highest regression risk) ships early in T2.

## Risks

| Risk | Impact | Mitigation |
|------|--------|------------|
| CLI prefix fix breaks existing overwrite behavior for `kokoro-*` files | High | `startswith("kokoro")` is a superset of `startswith("kokoro-")` — no behavior change for existing files. Verify with existing test_cli.py tests in T4 |
| Two skill files means double the creative content | Medium | Skills are simpler (routing + session management, not methodology-heavy). Knowledge files carry the depth |
| Router must reference skills that don't exist yet (Phase 2+) | Low | Design explicitly says MUST NOT reference Phase 2-4 skills by command name. Test for absence if needed |
