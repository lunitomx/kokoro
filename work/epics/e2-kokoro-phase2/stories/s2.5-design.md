# S2.5 Design — /kokoro-validate Skill

## Problem & Value

**Problem:** Entrepreneurs who completed their Lean Canvas, mapped Customer Forces, and ran interviews still lack a systematic method to design experiments that test their riskiest assumptions. Without a structured validation plan, they either build too much before testing or test the wrong things. Eduardo identifies "falling in love with the solution" as the cardinal sin of validation.

**Value:** A guided validation planning session structures the entrepreneur's next steps: identify risks from the Lean Canvas, design experiments with clear hypotheses, plan 3x3x3 sprints, and define traction metrics. This converts assumptions into evidence using the build-measure-learn cycle.

## Approach

Create `extension/.claude/commands/kokoro-validate.md` — a markdown prompt file executed when a user types `/kokoro-validate`. The prompt guides the user through entrepreneurial mindset principles, the 3x3x3 planning model, risk identification, experiment design, traction, and the Validation Plan Canvas, all in Eduardo's voice.

Follows the proven pattern from S2.2/S2.3 (PAT-L-001 + PAT-L-006): single markdown skill file with content validated by structural tests. No CLI changes needed — `_copy_dir_no_overwrite` already handles `commands/`.

The skill references `kokoro-phase2-validation.md` as knowledge context. Eduardo's coaching methodology applies: Proyector strategy (ask before guiding), powerful questions per section.

## Content Structure

| Section | Content | Key Concepts |
|---------|---------|-------------|
| Contexto | Purpose, experiment-driven mindset | Build-measure-learn, evidence over intuition |
| Estrategia del Proyector | Ask permission before guiding | Proyector strategy, invitation |
| 10 Principios | Entrepreneurial mindset principles | Amar el problema, traccion, 10X, evidencia |
| Modelo 3x3x3 | 3 years, 3 months, 3 weeks sprints | Vision anchoring, sprint cycles |
| Riesgos | 3 categories from Lean Canvas | Producto, Consumidor/Invitado, Mercado |
| Experimentos | Hypothesis template, qualitative vs quantitative | Hipotesis, metrica, umbral |
| Traccion | Velocity of monetizable value capture | Positive/negative indicators |
| Validation Plan Canvas | Leanstack 6-step process | Background through Follow-on Plans |
| Errores Comunes | Anti-patterns in validation | Solution love, Excel projections, measuring intentions |
| Resumen | Summary template with table | All key outputs in one view |

## Acceptance Criteria

**MUST:**
1. `kokoro-validate.md` exists in `extension/.claude/commands/`
2. Contains 3x3x3 planning model (3 anos, 3 meses, 3 semanas)
3. Contains all 3 risk categories (Producto, Consumidor/Invitado, Mercado)
4. Contains experiment design section with hypothesis and experiment concepts
5. Contains Validation Plan Canvas reference
6. Eduardo's voice present (invitado, creacion, inversion — all 3)
7. References knowledge file `kokoro-phase2-validation.md`
8. Anti-pattern warnings (solution love, Excel projections, measuring intentions)
9. Proyector strategy (permiso or invitacion)
10. Traction concept present
11. Mindset principles present
12. Summary table >= 4 rows
13. `kokoro init` copies the skill file to target `.claude/commands/`

**SHOULD:**
14. Experiment section has >= 2 guiding questions
15. Opening quote: "La velocidad de aprendizaje es la ventaja injusta."

**MUST NOT:**
16. Must not use generic marketing language (hacks, growth, monetizar)
17. Must not use "cliente" instead of "invitado", "precio" instead of "inversion", "producto" instead of "creacion"

## Decisions

| ID | Choice | Rationale | Alternatives |
|----|--------|-----------|--------------|
| D1 | Summarize 10 principles, not list all in full | Skill file is a coaching guide, not a textbook. Principles are reference, not the core exercise. | Full listing of all 10 with examples |
| D2 | 3 risk categories as separate subsections | Knowledge file treats each distinctly with its own examples. Combining would lose specificity. | Combined risk overview |
| D3 | Hypothesis template as structured format | Clear template reduces ambiguity in experiment design. | Free-form experiment description |
| D4 | Single markdown file following PAT-L-001 | Proven pattern from S2.2/S2.3. No CLI changes needed. | Multi-file skill |
| D5 | Test file mirrors test_forces.py structure | PAT-L-006: consistent test patterns reduce cognitive load. | New test structure |
