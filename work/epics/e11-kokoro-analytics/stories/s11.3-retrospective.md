## What Went Well

1. **Design examples drove implementation speed.** The 5 concrete examples
   in s11.3-design.md (platform-specific, cross-platform, not connected,
   server down, date range) served as the actual specification. Writing the
   skill file was mostly translating examples into prompt instructions. The
   estimated 30 min for Task 2 took 15 min.

2. **Established Kokoro skill pattern is mature.** This is the 5th Kokoro
   skill (after ads, connect, creative, audit) following the same structure:
   client resolution, MCP dispatch, Kokoro voice output. The pattern is
   well-understood and produces consistent results.

3. **TDD caught the mirror dependency.** The test fixture requires extension
   mirrors to exist for the `EXISTING_COMMANDS` check. Combining Tasks 2
   and 3 (GREEN + mirrors) into a single commit was the right call -- a
   separate mirror commit would have been green-to-green with no logical
   delta.

4. **QR caught a real semantic issue.** The "impressions" translation said
   "N personas vieron" (N people saw) but impressions count views, not unique
   people (that is reach). Changed to "las creaciones se mostraron N veces"
   which is semantically accurate. Good catch.

## What To Improve

1. **AR duplication questions remain open.** The Architecture Review flagged
   that the knowledge file duplicates some content from the skill file
   (metric glossary, platform table). The pragmatic answer is that knowledge
   files are reference data loaded separately from skill files, so some
   overlap is intentional. But this should be evaluated at the epic level --
   if S11.4 and S11.5 create more knowledge files, a shared reference
   pattern may emerge.

2. **Estimation for prompt-only skills is too generous.** At 50 min estimated
   vs 25 actual, the M-size estimate assumed more complexity than existed.
   Prompt-only skills with good design examples should be estimated at S
   (~30 min) regardless of the number of MCP integrations.

## Heutagogical Checkpoint

### 1. What did I learn?

The distinction between impressions and reach matters for metric translation.
"Personas que vieron" implies unique reach, while impressions count total
views (same person can generate multiple impressions). When translating
technical metrics to human language, semantic precision is more important
than natural-sounding language.

### 2. What would I change about the process?

Combine GREEN + mirror tasks by default when tests require mirrors. The
current plan template separates them as Task 2 (GREEN) and Task 3 (mirrors),
but if the test fixture validates mirror existence, they must be in the same
commit. The plan should reflect this dependency explicitly.

### 3. Are there improvements for the framework?

The plan template could include a "task merge" annotation when tasks have
commit-level dependencies. Something like `merge_with: Task 2` on Task 3
to signal they share a commit. This avoids the awkward "no separate commit"
note in the progress log.

### 4. What am I more capable of now?

Writing MCP-dispatching skills that route natural language queries to the
correct platform tools. The pattern (client resolution -> platform lookup ->
tool dispatch -> voice translation) is now a repeatable template. S11.4
(metric translation expansion) and S11.5 (scorecard) can build on this
foundation without re-inventing the routing logic.
