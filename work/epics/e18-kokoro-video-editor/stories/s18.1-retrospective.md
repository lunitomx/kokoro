## Summary

Evaluated OpenShorts (github.com/mutonby/openshorts) as a potential base for
the Kokoro Video Editor pipeline. The evaluation confirmed the approach
(Gemini for moment detection, face tracking for reframe, faster-whisper for
transcription) but found the codebase unsuitable for integration or forking
due to zero tests, no types, no package structure, and monolithic 2000+ line
files. Verdict: INSPIRATION-ONLY with 4 reference algorithms documented for
S18.3+ reimplementation.

## What Went Well

- **Pre-research saved time**: the OpenShorts evaluation was already performed
  before story formalization, so the story was purely documentation. This
  front-loaded the research risk and kept the story tight.
- **Clear verdict framework**: the integrate/fork/inspiration-only decision
  tree made the evaluation structured and actionable rather than open-ended.
- **Specific algorithm extraction**: identifying exactly 4 reusable algorithms
  with approximate line counts gives S18.3+ concrete starting points.
- **No code changes required**: documentation-only story meant zero risk of
  regressions. 581/581 tests remained green throughout.

## What to Improve

- **Scope doc had pre-research findings before plan existed**: the scope
  committed evaluation findings before the plan was written, which is slightly
  out of order. For future research stories, keep scope minimal and let the
  plan define what gets documented where.
- **No live testing of OpenShorts**: evaluation was based on code reading, not
  running the tool. For future tool evaluations, a hands-on test (even brief)
  would add confidence to the verdict.

## Heutagogical Checkpoint

### 1. What did I learn?

OpenShorts validates that LLM-based moment detection (sending full transcript
to Gemini with a detailed prompt) is a viable approach for identifying viral
clips. The SmoothedCameraman "heavy tripod" pattern (safe zone + slow pan +
fast re-frame) is an elegant solution to the face-tracking-for-reframe problem.
These are concrete algorithms worth reimplementing, not just vague inspiration.

### 2. What would I change about the process?

For documentation-only stories, the TDD cycle is N/A but the verification step
(T2: run tests) is still valuable as a regression check. The 2-task plan
(write doc + verify) was appropriate for this size. No process changes needed.

### 3. Are there improvements for the framework?

Research/evaluation stories could benefit from a lightweight evaluation template
with standard sections (overview, quality assessment, gap analysis, verdict,
extractable patterns). This would make evaluations more consistent across
different tools and libraries.

### 4. What am I more capable of now?

I have a concrete mental model of how face tracking for 9:16 reframe works
(SmoothedCameraman + SpeakerTracker), how LLM-based moment detection works
(full transcript to Gemini with structured prompt), and how subtitle generation
from word-level timestamps works. These inform S18.3+ implementation directly.
