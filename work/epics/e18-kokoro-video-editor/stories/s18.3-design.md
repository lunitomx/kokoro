# S18.3 Design — /kokoro-shorts

## Problem

/kokoro-cuts produces a cuts JSON with `start_sentence` and `end_sentence`
text boundaries, but ffmpeg needs numeric timestamps (seconds) to extract
video segments. There is no timestamp data in the cuts JSON or the plain-text
transcript. Without an automated resolution step, Eduardo would need to
manually find timestamps for each cut — defeating the purpose of the pipeline.

## Value

Completes Phase 1 of E18 (Milestone M1: Shorts Publicables). Eduardo goes
from a raw video to publish-ready shorts with zero manual editing. A 30-minute
video produces 5+ shorts in minutes instead of hours.

## Approach

/kokoro-shorts is a Claude Code skill (`.claude/commands/kokoro-shorts.md`)
that orchestrates three stages: timestamp resolution, ffmpeg extraction, and
output organization.

### Stage 1: Timestamp Resolution (D1)

The core technical challenge. Strategy: **re-transcribe with Whisper
verbose_json**.

1. Read the cuts JSON and extract `video_url`
2. Locate or download the audio file (reuse kokoro-listen's download if
   available at `/tmp/kokoro-listen/{video_id}.mp3`)
3. Call Whisper API with `response_format=verbose_json` — this returns
   segment-level timestamps with text
4. For each cut, fuzzy-match `start_sentence` against Whisper segments to
   find the start time; fuzzy-match `end_sentence` to find the end time
5. If a cut has explicit `start_time`/`end_time` fields, skip resolution
   for that cut (manual override)

**Why verbose_json re-transcription:**
- Whisper segments align naturally with speech boundaries
- The original transcript text matches Whisper output (same model, same audio)
- Cost is minimal (~$0.006/minute, so ~$0.02 for a 4-minute video)
- The audio file is already downloaded by kokoro-listen
- No manual timestamp entry, no external alignment tools

**Fuzzy matching algorithm:**
- Normalize both strings: lowercase, strip punctuation, collapse whitespace
- Use substring containment first (fast path)
- Fall back to longest common subsequence ratio with threshold >= 0.7
- Walk Whisper segments sequentially — start_sentence match gives start_time,
  end_sentence match gives end_time (use segment end timestamp)
- If no match found for a cut, report error and skip that cut (AC6)

### Stage 2: ffmpeg Extraction (D2)

For each resolved cut:

```bash
# Fast stream copy (no re-encoding)
ffmpeg -ss {start} -to {end} -i {input} -c copy {output_raw}

# Apply fade in/out (requires re-encoding the short segment)
ffmpeg -i {output_raw} \
  -vf "fade=t=in:st=0:d=0.5,fade=t=out:st={dur-0.5}:d=0.5" \
  -af "afade=t=in:st=0:d=0.5,afade=t=out:st={dur-0.5}:d=0.5" \
  -c:v libx264 -preset fast -crf 23 \
  -c:a aac -b:a 128k \
  {output_final}
```

Two-pass approach: first cut with stream copy (instant, no quality loss),
then apply fades with re-encoding (fast for 30-60s segments). This avoids
re-encoding the full video.

**Optional vertical reformat (9:16):**

```bash
ffmpeg -i {input} \
  -vf "crop=ih*9/16:ih,scale=1080:1920,fade=t=in:st=0:d=0.5,fade=t=out:st={dur-0.5}:d=0.5" \
  -af "afade=t=in:st=0:d=0.5,afade=t=out:st={dur-0.5}:d=0.5" \
  -c:v libx264 -preset fast -crf 23 \
  -c:a aac -b:a 128k \
  {output_final}
```

Center crop to 9:16 ratio, then scale to 1080x1920.

### Stage 3: Output Organization (D3)

Each short saved as `{output_dir}/{basename}-short-01.mp4`, `-short-02.mp4`, etc.

Summary JSON (`{basename}-shorts.json`):

```json
{
  "source_cuts": "/path/to/cuts.json",
  "source_video": "https://youtube.com/watch?v=...",
  "shorts_count": 3,
  "vertical": false,
  "shorts": [
    {
      "rank": 1,
      "file": "/tmp/kokoro-shorts/abc-short-01.mp4",
      "title": "Title from cuts JSON",
      "hook": "Hook text from cuts JSON",
      "start_time": 42.5,
      "end_time": 87.3,
      "duration": 44.8,
      "resolution_method": "whisper_fuzzy_match",
      "status": "success"
    },
    {
      "rank": 2,
      "file": null,
      "title": "Failed cut title",
      "start_time": null,
      "end_time": null,
      "duration": null,
      "resolution_method": "whisper_fuzzy_match",
      "status": "error",
      "error": "Could not match start_sentence in Whisper segments"
    }
  ]
}
```

### Invocation

```
/kokoro-shorts /tmp/kokoro-cuts/3zXGLEgl5L8-chatgpt-reuniones-cuts.json
```

Optional flags via instruction in the prompt:
- "vertical" or "9:16" in the user message triggers vertical reformat
- Output directory defaults to `/tmp/kokoro-shorts/` or client folder if
  resolved via /kokoro-open

### Input: Cuts JSON Schema (from /kokoro-cuts)

```json
{
  "source_transcript": "/path/to/transcript.txt",
  "video_title": "...",
  "video_duration": "3:43",
```

### Error Handling

- **Timestamp resolution failure for one cut**: skip that cut, report in
  summary JSON with error message, continue with remaining cuts (AC6)
- **Audio file not found locally**: download via yt-dlp (same as kokoro-listen)
- **ffmpeg not installed**: detect early, report clearly, abort
- **All cuts fail resolution**: report summary with all errors, no empty
  shorts directory created
- **Video file inaccessible (private/deleted)**: report error, abort gracefully

## Decisions

| ID | Choice | Rationale |
|----|--------|-----------|
| D1 | Re-transcribe with Whisper verbose_json for timestamp resolution | Whisper segments align with speech boundaries. Same model produces matching text. Cost ~$0.02/4min. No manual timestamps. Alternatives: manual timestamps (defeats automation), speech alignment tools (overengineered). |
| D2 | Two-pass ffmpeg: stream copy + re-encode for fades | Stream copy is instant and lossless for the cut. Re-encoding only the 30-60s segment for fades is fast. Single-pass re-encode of full video would be slow and unnecessary. |
| D3 | Summary JSON with per-short status (success/error) | Enables partial failure handling. Eduardo sees which shorts succeeded and which failed with reasons. Downstream skills (/kokoro-overlay) can filter by status. |

## Acceptance Criteria

- **AC1**: Reads cuts JSON and extracts video segments with ffmpeg
- **AC2**: Each segment has correct start/end based on Whisper timestamp resolution (within 2s tolerance)
- **AC3**: Fade in (0.5s) and fade out (0.5s) applied to each segment
- **AC4**: Output files named sequentially (`{basename}-short-01.mp4`, etc.) with summary JSON
- **AC5**: Optional 9:16 vertical reformat when requested (center crop to 1080x1920)
- **AC6**: Partial failure handling — one bad cut does not block others; errors reported in summary JSON

## Must Not

- Must not require manual timestamp entry (automation is the point)
- Must not re-download video if audio file already exists locally
- Must not re-encode the full source video (only cut segments)
- Must not fail silently — all errors visible in summary JSON
- Must not modify the input cuts JSON file
