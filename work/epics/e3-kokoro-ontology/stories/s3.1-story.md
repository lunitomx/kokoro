# Story: Ontology Schema

## User Story
As Kokoro's persistence layer,
I want Pydantic models that represent Eduardo's coaching domain concepts,
so that skill outputs can be serialized, validated, and persisted as typed nodes.

## Acceptance Criteria

### Scenario: All node types are defined
```gherkin
Given the ontology models module
When I inspect the available node types
Then there are at least 10 types: Segmento, Problema, PUV, Fuerza, Hipotesis, Experimento, Vision, OKR, Creacion, Metrica
```

### Scenario: All edge types are defined
```gherkin
Given the ontology models module
When I inspect the available edge types
Then there are at least 5 types: alimenta, valida, experimenta, mide, pertenece_a
```

### Scenario: CoachingState model validates
```gherkin
Given a CoachingState with nodes and edges
When I serialize to JSON and deserialize back
Then the round-trip produces an identical model
```

### Scenario: Pyright strict passes
```gherkin
Given the ontology models module
When I run pyright in strict mode
Then zero errors reported
```

## Examples (Specification by Example)

| Input | Action | Expected Output |
|-------|--------|-----------------|
| `NodeType.SEGMENTO` | Create CoachingNode | Valid node with type="segmento" |
| CoachingState with 3 nodes | `.model_dump_json()` | Valid JSON string |
| JSON string | `CoachingState.model_validate_json()` | Identical model |
| Node without required fields | Validate | Pydantic ValidationError |

## Notes
- Inspired by RaiSE's GraphNode pattern (open type system via subclass)
- Models only — no persistence logic (that's S3.2)
- Vocabulary in model docstrings must use Eduardo's terms (invitado, creacion, inversion)
