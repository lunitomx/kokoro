# S4.4 Design: Output Testing Spike

## Problem

Current tests verify skill file *content* (keywords, vocabulary, assertions).
Nothing tests the *output template structure* — the code blocks inside skills
that define what Claude should produce. If someone edits a skill and breaks
the output template, no test catches it.

## Approach: Template Structure Testing

Each coaching skill contains a closing output template in a fenced code block.
These templates define the structured output Claude should produce. We can
parse skill files, extract code blocks, and verify they contain required
structural elements.

**What we test (without Claude):**
1. Skills that define output templates have them in valid structure
2. Output templates contain required sections (headers, tables, next-step refs)
3. Cross-skill `/kokoro-X` references in "next step" sections point to real skills

**What we DON'T test:**
- Live Claude output (non-deterministic, expensive)
- Content quality of output (subjective)

## Design Decisions

1. **One test file:** `tests/test_output_structure.py` — separate from per-skill tests
2. **Parametrized:** `@pytest.mark.parametrize` over skills that have output templates
3. **Parse code blocks:** Extract fenced code blocks from markdown, test their structure
4. **Cross-ref validation:** All `/kokoro-X` references resolve to existing command files

## Viability Criteria

- **Viable** if: >=2 skills have testable output templates, tests catch real structural breaks
- **Not viable** if: output templates are too free-form to assert on meaningfully
