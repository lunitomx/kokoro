# Story: Output Testing Spike

## User Story
As a developer maintaining Kokoro skills,
I want a proof-of-concept for smoke-testing skill output structure,
so that regressions in skill output format are caught automatically.

## Acceptance Criteria

### Scenario: Skill output contains required sections
```gherkin
Given a Kokoro skill file (e.g., kokoro-diagnose.md)
When the skill template is parsed for output structure
Then required output sections are present (headers, markers, closing)
```

### Scenario: Spike documents approach viability
```gherkin
Given the PoC test implementation
When the approach is evaluated
Then a written assessment documents: viable/not viable, effort to scale, limitations
```

## Examples (Specification by Example)

| Input | Action | Expected Output |
|-------|--------|-----------------|
| kokoro-diagnose.md | Parse for output markers | Contains "## Diagnóstico", closing section |
| kokoro-canvas.md | Parse for output markers | Contains "## Canvas", Lean Canvas structure |

## Notes
- This is a timeboxed spike — 1 hour max
- Goal: determine if skill output structure can be tested without live Claude invocation
- If inconclusive, document learnings and defer
- PoC needs minimum 1 skill tested, ideally 2 (one E1, one E2)
