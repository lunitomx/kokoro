# Design: S15.1 — Session log model

## Approach

Create a single knowledge file that documents the session_log schema.
No code changes — this is a contract that /kokoro-open, /kokoro-close,
and all Kokoro skills reference when reading or writing session history.

## Schema: metadata["session_log"]

```json
{
  "session_log": [
    {
      "date": "2026-03-27",
      "type": "creative",
      "skill": "/kokoro-creative",
      "client_id": "crescer",
      "summary": "6 creativos Baby Balance — 2 publicos x 3 tamanos",
      "hallazgos": [
        "Publico mamas responde a dolor 'no se si mi bebe va bien'",
        "Fotos lifestyle > clinicas para segmento mamas",
        "JSON con text_critical_rules mejora acentos al 90%"
      ],
      "artifacts": [
        "campanas/meta-ads/creativo-01-mamas.txt",
        "campanas/meta-ads/creativo-02-profesionales.txt"
      ],
      "next_action": "Lanzar campana en Meta Ads con los creativos generados"
    }
  ]
}
```

### Field Definitions

| Field | Type | Required | Description |
|-------|------|:--------:|-------------|
| date | string (YYYY-MM-DD) | yes | Fecha de la sesion |
| type | string | yes | Tipo de trabajo: creative, ads, strategy, diagnostic, canvas, research, finance, launch |
| skill | string | no | Skill que genero la entrada (ej: /kokoro-creative) |
| client_id | string | yes | ID del invitado en el grafo |
| summary | string | yes | Que se hizo (1-2 lineas, concreto) |
| hallazgos | list[string] | no | Que se aprendio (insights del invitado, su publico, su mercado) |
| artifacts | list[string] | no | Paths relativos a clientes/{grupo}/ de archivos generados |
| next_action | string | no | Que hacer la proxima vez con este invitado |

### Design Decisions

**D1: Lista plana (no nested por fecha)**
Un array plano ordenado por fecha descendente. Simple de leer, simple
de escribir. No agrupar por mes o tipo — la lista es corta (max 20).

**D2: Max 20 entradas**
Al agregar la entrada 21, eliminar la mas antigua. 20 sesiones es
suficiente contexto sin inflar metadata. Si se necesita historial
completo en el futuro, se puede agregar archivado.

**D3: hallazgos como lista de strings (no objetos)**
Cada hallazgo es una frase corta y directa. No necesitan metadata
propia (tipo, confianza, etc.) — eso es over-engineering para
un historial conversacional.

**D4: artifacts como paths relativos a clientes/{grupo}/**
No paths absolutos. Relativos a la carpeta del grupo del invitado
para que sean portables.
